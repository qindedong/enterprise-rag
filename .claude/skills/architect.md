# 系统设计技能 (Architect)

## 描述
当用户需要进行系统架构设计、技术选型、模块划分或绘制架构图时，使用此技能提供专业的系统设计指导。

## 触发条件
- 用户提到"架构设计"、"系统设计"、"技术选型"、"模块划分"
- 用户需要设计新的子系统或重构现有架构
- 用户询问"怎么设计"、"如何架构"

## 设计原则

### 1. 需求分析先行
- 明确功能需求和非功能需求（性能、可扩展性、安全性）
- 识别核心业务实体和它们之间的关系
- 确定系统的边界和对外接口

### 2. 架构模式选择
根据项目特点选择合适的架构模式：
- **分层架构**：适合传统 Web 应用，层次分明
- **微服务架构**：适合大型、多团队协作项目
- **事件驱动架构**：适合异步处理、高并发场景
- **CQRS**：适合读写分离、复杂查询场景
- **管道-过滤器**：适合数据处理流水线

### 3. RAG 项目专用架构指南
本项目为**企业级知识库 RAG 系统**，设计时需重点关注：
- **文档处理管道**：解析 → 分块 → 向量化 → 索引
- **检索链路**：查询改写 → 多路召回 → 重排序 → 上下文组装
- **生成链路**：Prompt 模板 → LLM 调用 → 后处理 → 引用溯源
- **反馈闭环**：用户反馈 → 评估指标 → 策略优化

### 4. 模块划分原则
- **高内聚低耦合**：每个模块职责单一
- **接口先行**：先定义模块间的 API 契约
- **可替换性**：检索器、Embedding 模型、LLM 等核心组件应可插拔

### 5. 技术选型考量
- 优先选择团队熟悉的技术栈
- 考虑社区活跃度和文档完善度
- 评估性能指标是否能满足业务需求
- 关注许可证和长期维护成本

### 6. 输出格式
设计完成后，应输出：
1. **架构图**（使用 Mermaid 或文字描述）
2. **模块清单及职责说明**
3. **关键技术选型及理由**
4. **数据流向图**
5. **风险点和应对策略**
